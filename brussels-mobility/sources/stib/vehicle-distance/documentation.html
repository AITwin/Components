STIB/MIVB — Vehicle Distance

GET /stib/vehicle-distance

Provides the raw data from the STIB/MIVB proprietary API, reporting the distance traveled by each vehicle since its last stop. Useful for monitoring trip progress, congestion levels, or validating vehicle telemetry.

Description

This endpoint returns per-vehicle records indicating how far each vehicle has traveled since departing its previous stop. The data is refreshed continuously and can be queried for specific timestamps or intervals. Each entry links to a specific line, direction, and stop point (pointId).

Query Parameters

ParameterTypeRequiredDescription
timestamp integer No Unix timestamp (seconds since epoch). If not specified, the latest available data is returned; otherwise, the closest ≤ timestamp snapshot is returned.
start_timestamp integer No Start of the time window (epoch seconds).
end_timestamp integer No End of the time window (epoch seconds).
limit integer No Maximum number of records to return (for pagination).
offset integer No Record offset for pagination (default = 0).

Output Data Attributes

AttributeTypeDescription
directionIdstringDirection identifier indicating which directional pattern the vehicle is following—typically "0" or "1" corresponding to GTFS direction_id convention (0=outbound/one direction, 1=inbound/opposite direction). Actual geographic meaning varies by line: for radial routes like metro M1 (Gare de l'Ouest ↔ Stockel), direction 0 might be eastbound toward Stockel, direction 1 westbound; for circular routes, clockwise vs. counterclockwise. Critical for: associating distance measurements with correct route variant (same vehicle traveling opposite directions has different distance contexts), correlating with real-time position feeds (matching directionId to understand vehicle trajectory), and analyzing directional asymmetry (do vehicles travel different distances in one direction vs. other due to route variants, detours, or operational patterns?). Not meaningful in isolation—must interpret with lineId context.
distanceFromPointintegerDistance in meters that vehicle has traveled since departing the reference point (pointId)—typically the most recently departed stop. Measures inter-stop progress: value 0 indicates vehicle at stop (just departed or about to depart), values 50-300m suggest vehicle actively traveling between stops, values matching typical stop spacing (200-500m for Brussels surface transit, 400-800m for metro) indicate approaching next stop. Real-time progression metric enabling: arrival time estimation (distance remaining ÷ average speed = estimated time to next stop), service regularity monitoring (comparing distances for consecutive vehicles on same route reveals bunching—two vehicles both showing 100m from same point suggests inadequate spacing), congestion detection (vehicle showing same distanceFromPoint across consecutive observations indicates stalled/very slow progress), and route adherence validation (unexpectedly large distances suggest off-route operation or missed stop). Typical Brussels values: tram stops 250-400m apart, bus stops 200-350m, metro stops 500-1000m—distanceFromPoint values exceeding these ranges flag potential data quality issues, operational incidents, or express services skipping stops. Critical for passenger information systems calculating "vehicle approaching in X meters" and operations monitoring detecting service irregularities.
lineIdstringTransit line identifier—the route number/code passengers recognize (e.g., "1" for metro M1, "81" for tram Line 81, "95" for bus Line 95). Core reference linking distance measurements to specific STIB service lines. Enables: line-specific distance analysis (typical inter-stop distances for Line 81 tram vs. Line 1 metro), service quality monitoring by route (which lines show consistent progress vs. frequent stalls?), and cross-referencing with schedules/static data (joining lineId with GTFS routes.txt, stop_times.txt to contextualize distance within planned route geometry). Essential for operational dashboards filtering by line ("show all Line 1 vehicles and their inter-stop progress") and performance analytics aggregating distance metrics by route for comparative service evaluation across Brussels STIB network.
pointIdstringReference point identifier—typically the stop ID from which distanceFromPoint is measured (the vehicle's most recently departed stop). Establishes spatial reference for distance measurement: knowing pointId + distanceFromPoint enables precise vehicle location estimation (vehicle is X meters past stop Y along route path). Used for: next-stop identification (pointId is current/previous stop, next stop determined by route sequence), arrival prediction (distance from pointId to next stop minus distanceFromPoint = remaining distance), and geographic contextualization (joining pointId with stops dataset retrieves stop name, coordinates, enabling "vehicle 250m past Arts-Loi metro toward Stockel" passenger-facing messages). May also represent other measurement points (route waypoints, timing points, virtual reference locations) depending on STIB telemetry implementation. The ~3.1 million historical records spanning Feb 2023-Aug 2025 indicate high-frequency real-time data collection—multiple distance updates per vehicle per trip enabling fine-grained trip progression monitoring essential for Brussels transit operations and passenger information accuracy.

Example Application Scenarios

  • Correlating distance with trip completion or punctuality metrics.

Operational Context for Brussels Transit Ecosystem

The STIB Vehicle Distance endpoint provides real-time trip progression telemetry by tracking how far each vehicle has traveled since its last reference point—essential for passenger information systems generating "next tram arriving in 150 meters" displays and mobile app notifications enabling precise wait-time estimates at Brussels stops. For operations controllers, distance progression rates reveal service quality issues: vehicles showing minimal distance change across consecutive observations flag stalled operations (traffic congestion, mechanical issues, driver break overruns), while comparing distances between consecutive vehicles on the same route detects bunching (two buses both 100m from same stop = inadequate spacing requiring dispatch intervention). The dataset's 3.1+ million records spanning February 2023 through August 2025 enable performance analysts to establish line-specific baselines (typical inter-stop distances and travel patterns for tram Line 81 vs. metro M1), identify congestion corridors through slow distance accumulation rates (chronic delays on specific segments indicating infrastructure bottlenecks or traffic interference), and validate GPS positioning accuracy by comparing odometer-based distanceFromPoint with GPS-calculated distances (persistent discrepancies suggest sensor calibration issues). For software developers, this endpoint integrates with GTFS stop sequences and route shapes to provide complete vehicle location context: joining pointId with stops.txt retrieves geographic coordinates and stop names, calculating "distance remaining to next stop" enables arrival predictions, and correlating with vehicle-position GPS feeds validates positional accuracy across telemetry systems. Urban mobility researchers leverage historical distance patterns to quantify service reliability (consistency of inter-stop travel times and distances), evaluate infrastructure interventions (did dedicated bus lanes reduce distance-to-time variability?), and model passenger experience (frequent vehicle stalls = poor service perception). The simple four-field structure (lineId, directionId, pointId, distanceFromPoint) provides fundamental building blocks for Brussels transit monitoring—when a vehicle on Line 81 direction 0 shows 250 meters from Arts-Loi stop, operators know the tram is mid-journey toward Montgomery, passengers can estimate 30-second arrival times, and quality systems confirm normal progression rather than service disruption.
